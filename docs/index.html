EvoHarness

Evolution Tree Search for Agent Harness Optimization

Paradigm Automated Research Hackathon — April 9, 2026

Motivation Foundation Contributions The Loop Experiment Reproduce

An LLM agent's performance depends not only on the underlying model but on the harness—the surrounding code that constructs prompts, defines tools, manages context, and orchestrates execution. Recent work has shown that these harnesses can be optimized automatically: Meta-Harness (Lee et al., 2026) demonstrated that giving a coding agent full access to execution traces produces dramatically better harness proposals than score-only feedback (50% vs 35%). better-harness (LangChain) introduced structured surface decomposition and train/holdout separation. AutoHarness (Google DeepMind) applied Thompson sampling tree search to synthesize game-playing harnesses.

However, applying these ideas to complex agent benchmarks remains inefficient. On TerminalBench-2—89 challenging command-line tasks—Meta-Harness's optimization loop failed 7 out of 8 times, producing regressions or neutral results. After ~$2,400 in compute and 8 iterations, only one useful change was found (+1.7%). The core problem is not the proposer or the trace access—it is the search strategy.

We present EvoHarness—a framework that replaces linear hill climbing with tree-structured search over harness variants. Our key contributions:

  1. Branch: Multiple proposers explore different surfaces in parallel, creating a tree of variant branches selected by Thompson sampling.
  2. Screen: Cheap pre-screening (3 tasks, ~$9) filters bad proposals before expensive full evaluation (~$267), reducing waste by 30x.
  3. Merge: Accepted branches are systematically merged to test whether orthogonal improvements compose.

Supporting these, we introduce surface fragility tracking (automatically learning which harness components are safe to modify) and a research notebook (persistent memory of dead ends that prevents repeating failed approaches across iterations).

In our first experiment on TerminalBench-2, a single iteration with one proposer flipped 9 previously-failing tasks to passing—more improvement than Meta-Harness achieved in 8 iterations. A second parallel branch found an additional orthogonal improvement, confirming that the changes compose.

Why Meta-Harness Failed on TerminalBench-2

Meta-Harness search trajectory (8 iterations):

  Iter 1: bundled prompt + structural fixes    → REGRESSION (58.9%)
  Iter 2: different bundle                     → REGRESSION (57.8%)
  Iter 3: (diagnosed confound)
  Iter 4: isolated completion logic fix        → REGRESSION
  Iter 5: another completion logic fix         → REGRESSION
  Iter 6: yet another completion logic fix     → REGRESSION
  Iter 7: additive env bootstrapping           → ACCEPTED (+1.7%)
  Iter 8: compose with prior                   → NEUTRAL

  8 iterations. 1 useful change. ~$2,400 spent.

Three structural problems:

1. No memory of dead ends. Iterations 4, 5, and 6 all tried modifying completion_logic. All regressed. The proposer had no record of prior failures, so it repeated the same mistake three times.
Our fix: Surface fragility tracking + dead ends notebook. Each surface has an automatically-updated fragility score. After one rejected edit, the score rises. After three, it's marked HIGH risk and the proposer sees a warning. The research notebook records why each approach failed—not just that it was rejected—so the proposer avoids the same class of mistake on any surface.
2. No pre-screening. Every proposal—good or bad—required a full evaluation on all 89 tasks (~$267). The cost of a bad idea was the same as a good one.
Our fix: 3-task pre-screening. Before committing to a full eval, run the proposal on just 3 failing tasks (~$9). If none flip, prune immediately. Cost of a rejected proposal drops from $267 to $9—a 30x reduction in wasted compute.
3. Linear chain blocks exploration. One proposal at a time, sequentially. No ability to explore multiple hypotheses in parallel or compose orthogonal improvements from different branches.
Our fix: Tree search with parallel proposals and branch merging. Multiple Claude Code proposers run in parallel, each targeting a different surface. Accepted changes become branches that can be independently extended or merged to test if improvements compose. Thompson sampling balances exploring new branches vs extending promising ones.

The Iteration Loop

Each iteration follows a strict process designed to prevent false positives and wasted compute:

Step 1: PROPOSE
  Claude Code agent reads failing traces + notebook (dead ends, surface risk)
  → Proposes ONE change to ONE surface
  → Avoids HIGH-fragility surfaces and approaches listed in dead_ends.md

Step 2: PRESCREEN ($9, ~5 min)
  Run on 3 randomly-selected failing tasks, default resources
  → 0/3 flip → PRUNE immediately. No further cost.
  → 1+ flip → proceed

Step 3: EVAL FAILING + CANARIES ($75, ~20 min)
  Run on all ~20 failing tasks + 5 randomly-selected passing tasks
  → Flips: how many failing tasks now pass?
  → Canaries: did any passing tasks regress?

Step 4: GATEKEEPER
  ACCEPT if: pass_count improves AND regressions ≤ 1
  REJECT otherwise

Step 5: UPDATE NOTEBOOK
  If accepted:
    → findings.md: "verification protocol flipped 9 tasks"
    → Surface fragility stays low
    → New branch in the search tree

  If rejected:
    → dead_ends.md: "error recovery in prompt caused 3 regressions on X, Y, Z"
    → Surface fragility increases (0.0 → 0.5 → 0.75 → HIGH)
    → Next proposer sees the warning, avoids same approach

The fragility tracking is critical. When Meta-Harness's proposer tried modifying completion_logic three times, each time regressing, there was no mechanism to stop it. In our framework, after the first rejection the fragility rises to 0.5 (MEDIUM). After the second, 0.75 (HIGH). The proposer sees "completion_logic [HIGH risk, fragility=0.75]" in the surface manifest and switches to a different surface.

The dead ends notebook goes further—it records why the approach failed, not just that it was rejected. This prevents the proposer from trying the same class of change on a different surface. "Error recovery guidance causes timeout" is a lesson that applies to system_prompt and tool_definitions alike.

Reproducing

# 1. Baseline eval (all 89 tasks, 1 trial)
harbor run -d terminal-bench@2.0 -m anthropic/claude-opus-4-6 -e modal \
  --agent-import-path agent:AgentHarness -k 1 -n 89 \
  -o jobs/baseline -y --env-file .env

# 2. Import traces into the framework
python -m meta import jobs/baseline/baseline --output runs/experiment

# 3. Run a proposer (investigates traces, writes proposal)
python -m meta.run_iteration --experiment-dir runs/experiment \
  --iteration 1 --parent-variant baseline \
  --proposer-model sonnet --eval-env modal

# 4. Check which tasks flipped
for d in runs/experiment/iter_001/harbor_jobs/iter_001/*/; do
  echo "$(basename $d | sed s/__.*//): $(cat $d/verifier/reward.txt 2>/dev/null)"
done

# 5. Validate best harness on leaderboard conditions
cd runs/experiment/iter_001_branch_001/harness
harbor run -d terminal-bench@2.0 -m anthropic/claude-opus-4-6 -k 5
